# Spec 001 — Upcoming Trips Summary Dashboard

Status: Draft
Last updated: 2026-06-14
Production URL: https://trips-dashboard-theta.vercel.app

## 1. Goal

Create an authenticated, read-only Vercel dashboard that lists Danny's upcoming trips using the same operational truth as the Hermes `travel-planner` skill.

The first delivery slice is deliberately modest: a visually polished summary list of current and future trips, inspired by the existing meals dashboard theme and layout. The important part is the data boundary: trip data must be generated from private Hermes travel state and must not be committed to the public GitHub repository or placed in unauthenticated static assets.

## 2. Non-goals for this slice

- No trip editing from the dashboard.
- No WhatsApp, SMS, Calendar, Gmail, or Home Assistant actions from the dashboard.
- No live monitoring controls.
- No raw evidence body display.
- No precise home address, exact home coordinates, raw private calendar descriptions, raw email contents, OAuth tokens, contact exports, or WhatsApp history.
- No public static `real-data.ts`, `data.json`, or equivalent generated private data committed to git.

## 3. Users and access

| User | Access |
|---|---|
| Danny | View all upcoming trip summaries after authentication. |
| Future household/admin users | May be allowed through OIDC groups/claims later. |
| Anonymous visitor | Must see no trip data. Redirect to sign-in once OIDC is implemented. |

Authentication is specified separately in [`002-oidc-auth.md`](002-oidc-auth.md). Until that is implemented, the dashboard must use mock/sample data only, or remain disabled for production private data.

## 4. Source of truth

The dashboard reads a derived dashboard projection from the existing travel-planner state:

```text
~/.hermes/workspace/travel-planner/
├── trips/<trip_id>.yaml                 # canonical private trip state
├── discovery-runs/<timestamp>.json      # private discovery evidence and scoring
└── monitoring/<trip_id>.json            # private runtime monitoring lifecycle
```

The projection must be generated by the travel-planner workflow or a companion sync script after relevant travel runs:

- discovery/reconciliation runs
- planning/refinement runs
- monitoring lifecycle/status updates when they materially change the summary
- manual targeted reconciliation runs

The projection is not the canonical record. It is a display-safe summary derived from canonical private state.

## 5. Data flow

### 5.1 Target architecture

```text
Hermes travel-planner runs
  ├─ read/update private trip YAML + monitoring JSON
  ├─ build display-safe dashboard projection
  └─ POST projection to authenticated private dashboard ingestion endpoint
         │
         ▼
Private runtime store on Vercel side
  - Vercel KV/Postgres or equivalent authenticated storage
  - protected by Vercel environment secrets
  - never committed to git
         │
         ▼
Next.js dashboard server route/page
  - requires OIDC session
  - reads projection server-side
  - renders upcoming trips list
```

### 5.2 Ingestion requirements

- `POST /api/trips/sync` accepts the dashboard projection.
- The endpoint must require a server-side sync secret, HMAC signature, mTLS-equivalent gateway control, or other non-user ingestion credential.
- The sync credential must live only in runtime environment variables/secrets, never source files.
- Ingestion must replace the current projection atomically, or version it with `generated_at` and expose only the latest valid version.
- Failed syncs must not clear the last known good dashboard projection.
- The sync process should be invoked as part of all trip-producing workflows rather than relying on a separate dashboard-only cron.

### 5.3 Rendering requirements

- Dashboard pages and data APIs must read projection data server-side after OIDC session validation.
- No private trip payload may be emitted into `.next/static`, public JSON files, public assets, or source-controlled generated modules.
- Client-side data may only be data already authorised for the authenticated session.

## 6. Upcoming trip inclusion rules

Display trips that satisfy all of the following:

- `trip_window.start` is today or in the future, or the trip is currently active today.
- Status is one of: `candidate`, `confirmed`, `planned`, `monitoring`, `active`.
- Exclude historical completed trips by default.
- Exclude cancelled trips by default unless a future action/question is still attached.
- Sort by start date/time ascending.

This should mirror the travel-planner `trip index` behaviour: concise, current/future-focused, and suitable for choosing the next trip to work on.

## 7. Dashboard projection contract

The first implementation should define a TypeScript contract similar to this shape:

```ts
export interface TripsDashboardProjection {
  generatedAt: string;          // ISO timestamp
  source: 'travel-planner';
  schemaVersion: 1;
  trips: TripSummary[];
  summary: TripsSummary;
}

export interface TripsSummary {
  upcomingCount: number;
  activeCount: number;
  needsPlanningCount: number;
  monitoringEnabledCount: number;
  nextTripId?: string;
}

export interface TripSummary {
  tripId: string;
  title: string;
  status: 'candidate' | 'confirmed' | 'planned' | 'monitoring' | 'active' | 'completed' | 'cancelled';
  start: string;                // ISO date or datetime
  end: string;                  // ISO date or datetime
  destinationLabel?: string;    // town/city/venue label; no precise home address
  locations: TripLocationSummary[];
  travellers: string[];         // e.g. ['Danny'], ['Family']; no raw attendee emails
  planningStatus: 'needs_info' | 'ready_to_plan' | 'planned' | 'monitorable' | 'unknown';
  monitoring: {
    enabled: boolean;
    active: boolean;
    lastCheckAt?: string;
    nextCheckHint?: string;
    coverageStatus?: 'not_configured' | 'configured' | 'active' | 'degraded' | 'unknown';
  };
  nextAction?: {
    label: string;
    severity: 'info' | 'attention' | 'blocked';
  };
  legs: TripLegSummary[];
  sourceUpdatedAt: string;
}

export interface TripLocationSummary {
  label: string;
  precision: 'venue' | 'town' | 'city' | 'region' | 'unknown';
  role: 'destination' | 'stop' | 'origin_area' | 'context';
}

export interface TripLegSummary {
  legId: string;
  sequence: number;
  fromLabel: string;
  toLabel: string;
  departAfter?: string;
  arriveBy?: string;
  transportMode?: 'driving' | 'train' | 'public_transport' | 'taxi' | 'walking' | 'cycling' | 'flight' | 'unknown';
  monitoringChecks: string[];
}
```

### 7.1 Redaction rules

The projection must not contain:

- precise home address or coordinates
- raw calendar descriptions
- raw email bodies or snippets that include booking references beyond short display-safe labels
- raw WhatsApp content
- phone numbers, email addresses, OAuth IDs, calendar IDs, event IDs, or contact IDs
- API credentials or provider tokens
- unredacted work details

Allowed summary fields:

- high-level trip title
- town/city/venue labels when already safe for user-facing trip reports
- dates/times
- coarse status, planning readiness, monitoring flags
- leg labels and selected transport mode
- concise next action text written for Danny

## 8. Visual design

Take inspiration from the meals dashboard rather than inventing a separate visual language:

- Dark-first theme with light-mode toggle.
- Use the meals dashboard CSS token family where practical: `--bg-primary`, `--bg-secondary`, `--bg-tertiary`, `--border-color`, `--text-primary`, `--text-secondary`, `--accent-emerald`, `--accent-amber`, `--accent-rose`, `--accent-blue`, `--accent-purple`, `--accent-cyan`.
- Sticky compact header with dashboard title and theme toggle.
- Summary metric cards across the top.
- Rounded cards, subtle borders, compact typography, clear status colours.
- Mobile-first single-column list; desktop two-column layout may be introduced once detail panels exist.

### 8.1 Initial page layout

```text
Header
  🧭 Trips Dashboard                         theme toggle

Metric cards
  Upcoming | Active | Needs planning | Monitoring enabled | Next trip

Main list
  Trip card
    title + status pill
    date range + destination
    travellers
    planning readiness
    monitoring status
    next action / blocker if any
    compact legs summary
```

### 8.2 Status colour mapping

| Status | Colour intent |
|---|---|
| `active` | emerald / blue accent, visually prominent |
| `monitoring` | blue/cyan |
| `planned` | emerald |
| `confirmed` / `ready_to_plan` | amber/info |
| `candidate` / `needs_info` | amber attention |
| `cancelled` / blocked | rose, normally hidden unless action remains |
| degraded monitoring coverage | rose/amber attention depending severity |

## 9. Empty, loading, and error states

- Empty upcoming trips: show a calm empty state: "No upcoming trips found" plus last sync timestamp if available.
- No authenticated session: OIDC sign-in page once auth is implemented.
- No private data store configured: show an admin-safe configuration error after sign-in, never mock private details.
- Sync stale: show `generatedAt` and warning if older than a configured threshold, for example 24 hours.
- Sync failed but previous data exists: continue showing previous data with a stale warning.

## 10. Operational invariants

1. The dashboard projection is generated from travel-planner canonical state; the dashboard does not reinterpret raw Calendar/Gmail/WhatsApp evidence.
2. Public git contains source code, specs, mock fixtures, and tests only — never live trip data.
3. Private data APIs and pages require authentication before returning trip data.
4. Vercel static build output must not contain real trip data.
5. Sync is part of trip lifecycle runs, so discovery and planning updates become visible without manual dashboard edits.
6. Failed dashboard sync must not block travel-planner discovery/planning itself; it should produce an actionable warning.

## 11. Acceptance scenarios

### SC-001 — Upcoming trips list

Given a private projection with three future trips, when Danny opens the authenticated dashboard, then he sees all three sorted chronologically with status, date range, destination label, planning readiness, monitoring status, and next action.

### SC-002 — Cancelled trips hidden

Given a cancelled historical trip and a completed past trip, when the dashboard renders, then neither appears in the default upcoming list.

### SC-003 — Active trip retained today

Given a trip that started yesterday and is active today, when the dashboard renders, then it appears above or within the chronological list with an `active` status.

### SC-004 — Private data not committed

Given a sync has run, when `git status` and the repository diff are inspected, then no live trip JSON/TypeScript data file is present for commit.

### SC-005 — Static private-data guard

Given a production build has completed, when the build output is scanned for configured private sentinels such as real trip IDs or destination strings, then no match appears under static/public assets.

### SC-006 — Unauthenticated visitor

Given OIDC is implemented and an anonymous visitor opens `/`, then no trip data is returned and the visitor is redirected to sign-in.

## 12. Implementation notes for later

- Reuse structural ideas from `meals-dashboard`: Next.js app router, TypeScript contracts under `lib/`, CSS variables in `app/globals.css`, `ThemeProvider`, `ThemeToggle`, compact metric cards, and a custom Authentik sign-in page.
- Unlike meals-dashboard's generated `lib/real-data.ts`, trips-dashboard must not write live data into source-controlled modules because the repository is public.
- Add a small script analogous to `scripts/scan-static-private-data.mjs` that accepts sentinel strings from environment/config and fails if private data lands in static output.
- Prefer mocked fixture data under `fixtures/` or test files only when clearly artificial and non-private.
- The future sync script should live either with the travel-planner skill or in this repo as source only; runtime credentials and generated payloads stay outside git.
